12 /**
13  * Why make Easy_ref_t RAII-less?
14  *
15  * It would simplify design of Easy_ref_t, since it doesn't need to
16  * implement RAII-part logic.
17  *
18  * It would also make it easier to use member function of Easy_ref_t in Multi_t,
19  * provided that libcurl callback provides CURL* and that option CURLOPT_PRIVATE
20  * which enables storing any object, isn't support until 7.10.3.
21  *
22  * Easy_ref_t's member function cannot be called in multiple threads simultaneously.
23  *
24  * PERSISTENT CONNECTIONS
25  *
26  * Persistent connections means that libcurl can re-use the same connection for several transfers,
27  * if the conditions are right.
28  *
29  * libcurl will always attempt to use persistent connections.
30  *
31  * Whenever you use Easy_ref_t::perform or Multi::perform/Multi::socket_action,
32  * libcurl will attempt to use an existing connection to do the transfer, and if none exists it'll
33  * open a new one that will be subject for re-use on a possible following call to these functions.
34  *
35  * To allow libcurl to take full advantage of persistent connections, you should do
36  * as many of your file transfers as possible using the same handle.
37  *
38  * If you use the easy interface, and the Easy_t get destroyed, all the possibly
39  * open connections held by libcurl will be closed and forgotten.
40  *
41  * When you've created a multi handle and are using the multi interface, the connection pool is
42  * instead kept in the multi handle so closing and creating new easy handles to do transfers
43  * will not affect them.
44  * Instead all added easy handles can take advantage of the single shared pool.
45  *
46  * It can also be archieved by using curl::Share_base or curl::Share and
47  * enable_sharing(Share_base::Options::connection_cache).
48  */
